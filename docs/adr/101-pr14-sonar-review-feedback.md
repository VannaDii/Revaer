# PR 14 Sonar Review Feedback Resolution

- Status: Accepted
- Date: 2026-03-28
- Context:
  - Motivation:
    - Address the remaining PR feedback on the Sonar workflow branch before merge.
    - Keep the coverage gate cost stable in `just ci` while still producing Sonar upload artifacts.
    - Align the workflow and documentation with the repository's SonarCloud deployment.
  - Constraints:
    - Repository policy requires all operational paths to flow through `just` recipes.
    - Forked pull requests cannot access the repository Sonar token.
    - The compile database emitted by `build.rs` must match the actual target directory used by `just sonar-compile-db`.
- Decision:
  - Add `cov-tools` as the shared `cargo-llvm-cov` bootstrap recipe, keep `just cov` as the per-crate 90% gate, and add `just cov-lcov` for workspace LCOV and HTML artifacts.
  - Guard the Sonar workflow job so forked pull requests skip the secret-backed scan path.
  - Rename workflow and ADR language from SonarQube to SonarCloud where that distinction matters.
  - Derive the compile database object output from `CARGO_TARGET_DIR`, falling back to the workspace `target/` directory only when the variable is unset.
  - Alternatives considered:
    - Leaving LCOV generation inside `just cov`: rejected because it doubles the default coverage runtime in `just ci`.
    - Switching to `pull_request_target`: rejected because the current need is to avoid secret failures on forks, not to increase trusted execution scope.
    - Keeping SonarQube naming: rejected because the repository is configured for SonarCloud and the review feedback correctly identified the mismatch.
- Consequences:
  - Positive outcomes:
    - `just ci` keeps the existing coverage gate behavior without the extra workspace LCOV run.
    - Sonar artifact generation moves to an explicit `just cov-lcov` path that the workflow can call directly.
    - Forked pull requests no longer fail the Sonar job due to missing secrets.
    - `compile_commands.json` now reflects the real `CARGO_TARGET_DIR` used by the sonar compile recipe.
  - Risks and trade-offs:
    - SonarCloud coverage artifacts are now generated by a distinct recipe, so workflow and documentation must stay aligned with that split.
    - Fork PRs skip SonarCloud analysis entirely, which trades off contributor ergonomics against unavailable secrets.
- Follow-up:
  - Test coverage summary:
    - Validate with `just fmt`, `just lint`, `just udeps`, `just audit`, `just deny`, `just test`, `just cov`, `just cov-lcov`, `just build-release`, and `just ui-e2e`.
  - Observability updates:
    - No runtime telemetry changes; this task only adjusts developer tooling and CI workflow behavior.
  - Risk and rollback plan:
    - Roll back by removing `cov-lcov`, the fork guard, and the target-dir-aware compile command emission if they cause unexpected CI behavior.
  - Dependency rationale:
    - No new dependencies were added.
